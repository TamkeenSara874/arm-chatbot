# Architecture Notes

See [`architecture_diagram.html`](architecture_diagram.html) for the full visual diagram. This document covers the key design decisions and tradeoffs in bullet form.

## Stack and why

- **FastAPI + Postgres + Qdrant + Redis + React/Vite.** Async throughout since this is an I/O-bound AI system (LLM calls, vector search, DB queries) — see `docs/architecture_diagram.html` for the request-level flow.
- **Qdrant hybrid search (dense + native sparse), not a separate BM25 library.** Dense (`text-embedding-3-large`, 3072-dim) and sparse (fastembed `Qdrant/bm25`) vectors are stored as two named vectors on the same collection, fused server-side via `query_points()` with RRF. This replaced an earlier in-process `rank-bm25` index, which had a hard single-worker constraint (each worker held its own index, giving non-deterministic rankings under `--workers > 1`). The native approach removes that constraint entirely — sparse ranking is now handled by Qdrant itself, consistent across any number of API workers.
- **Groq (`llama-3.3-70b-versatile`) for decomposition, OpenAI for generation.** Groq's free tier is fast enough for the classification/decomposition step (which doesn't need top-tier reasoning) and costs nothing; the two generation models (`gpt-4o-mini` simple, `gpt-4.1` complex) are the ones actually producing the user-facing answer, where quality matters more than cost per call.
- **Cross-encoder reranking (`cross-encoder/ms-marco-MiniLM-L6-v2`), warmed up at startup.** Both the reranker and the sparse-embedding model are loaded during FastAPI's `lifespan()` startup, which blocks until complete — `uvicorn` does not start accepting connections until warmup finishes. This means the model load happens once per container start, never on a user's first chat message. Confirmed via `/health/ready`'s `reranker_warmed_up`/`sparse_embedder_warmed_up` fields. Originally `BAAI/bge-reranker-base` (278M params); live per-stage timing (`retrieval_breakdown` log) showed it taking 13-22s of CPU inference for just 24-30 candidates — the dominant end-to-end latency cost by a wide margin — so it was replaced with MiniLM-L6-v2 (~22M params), a much smaller model purpose-built for reranking.
- **Reranker: ONNX int8 quantization + a tuned batch size, in that order of discovery.** Live traces still showed reranking as ~45% of total request latency even after the MiniLM swap. Quantizing to ONNX int8 looked like the fix on a synthetic benchmark, but live traffic showed no improvement — root-caused to real review text having far more length variance than the benchmark's short synthetic pairs; `CrossEncoder.predict()`'s default `batch_size=32` meant every request's candidates (always ≤20) landed in one padding-dominated batch, since the model's own length-sort only helps *across* multiple batches. Setting `reranker_batch_size=8` (small enough to split into length-homogeneous sub-batches, large enough not to fragment unnecessarily) fixed the real problem and *also* unlocked quantization's real ~15-20% additional win on top of it. Both are on by default (`RERANKER_ONNX_QUANTIZED`, `RERANKER_BATCH_SIZE` — env-overridable, rollback needs no code change).

## Chunking strategy

- Reviews ≤256 tokens become a single chunk; longer reviews use a sliding window (32-token overlap) on sentence boundaries (NLTK `sent_tokenize`), not raw character splitting. Reviews are short-form text, so most stay as single chunks — chunking exists for the minority of long, multi-topic reviews where returning a whole review as one chunk would dilute a specific complaint/praise with unrelated content.

## Guardrail and intent design

- Guardrail is a **cheap, code-only lookup**, not an LLM call: decomposition classifies intent once (Groq), then `check_guardrail(intent)` does a dict lookup. Guardrailed intents (`out_of_scope`, `manipulation_request`, `multi_location`, `allergen`) return a canned decline with zero additional cost.
- **`ui_question` vs `report_howto` vs `conversation_recall` are deliberately split.** Generic platform/dashboard navigation questions ("how do I log in?", "why is this icon red?") redirect to AIO's CareBot, since this bot has no visibility into the platform's UI. "How do I download my report?" is answered directly with real instructions (Report button → Download PDF) — it's a feature this app actually has, so declining or redirecting it would be a worse user experience than just answering. "What did I ask before?" is neither — CareBot has no visibility into *this chatbot's own* conversation history either — so it gets its own `conversation_recall` intent and a retrieval-free answer path (see "Decomposition robustness" below).
- `DecomposedQuery.intent` is a `Literal[...]` of the exact intent set the decomposition prompt documents, not a bare `str`. A hallucinated/malformed intent fails Pydantic validation, which triggers `decompose_query()`'s retry-then-safe-fallback (defaults to `factual`) instead of silently bypassing the guardrail because the intent string didn't match any known guardrail key.

## Evidence ranking and hallucination control

- Composite score = `injection_penalty * (w_rrf*relevance + w_recency*recency_decay + w_rating*effective_rating)`, where `relevance` is the cross-encoder reranker's sigmoid-normalized score (Qdrant's own dense+sparse RRF fusion already happened server-side inside `hybrid_retrieve()`; nothing downstream re-runs RRF over the reranked list — the `w_rrf` weight name is historical). `effective_rating` uses the pre-computed sentiment label instead of the raw star rating whenever they disagree (`sentiment_rating_agree=False`), so a sarcastic 5-star complaint doesn't rank as if it were genuinely positive.
- **Hard hallucination gate:** if retrieval returns zero evidence, the pipeline never calls the generation LLM at all — it returns a canned "I couldn't find any reviews matching that" answer. This is a stronger guarantee than prompt instructions alone ("never fabricate" is a soft constraint the model can still violate under real traffic).
- **Groundedness heuristic (`src/core/groundedness.py`):** a regex-based, code-only check that flags an answer if it states a review/mention count higher than the number of evidence chunks actually retrieved (excluding star ratings and years, which coincidentally look like counts). This runs on every response with zero added cost/latency and discounts `confidence` when triggered — it's a proxy for "did the model overclaim," not a full accuracy judge.
- **Compound queries** (a countable half + a generative half, e.g. "how many positive reviews do I have, and how can I improve?") always route through the complex prompt with the DB-exact count passed in verbatim (`exact_count`), so the model states a real number instead of trying to (mis)count evidence chunks itself.

## Cost and latency controls

- **Real token-based costs, not estimates.** Every LLM/embedding call requests real usage (`stream_options={"include_usage": true}` for streaming, `response.usage` for non-streaming) and reports it via `RequestTrace.record_tokens()` / `IngestTrace.record_entity_tokens()` / `record_embedding_tokens()`. `cost_usd` in every trace and every ingest run reflects actual provider-billed tokens against `MODEL_COSTS_USD_PER_1M`, not a character-count approximation.
- **Fast paths bypass generation entirely:** `count_query` intent hits a direct Postgres `COUNT(*)` (target <100ms, $0); the guardrail path costs $0 beyond the one decomposition call; a cache hit costs $0 and skips the whole pipeline; `conversation_recall` skips review retrieval/ranking entirely and only pays for one cheap `gpt-4o-mini` call over session context.
- **Response cache, two tiers:** an exact-text tier (SHA-256(query text) → Redis, TTL 24h, checked before decomposition -- cheapest, catches a literal repeat with zero LLM cost) and a semantic tier (embeds the decomposed/rephrased query, searches a small `chat_cache` Qdrant collection filtered by `restaurant_id`, accepts a hit at cosine similarity ≥ 0.95, then fetches the actual response from Redis) that catches paraphrases of an already-answered question. The semantic tier's embedding is reused for retrieval on a miss rather than computed twice. Review data only changes on re-ingestion (not continuously), and ingestion already calls `cache.invalidate_restaurant()` to bust the whole restaurant's cache on new data, so a long TTL is close to pure cost savings. Corrections additionally call `cache.invalidate_query()` to bust just that one query's exact-tier entry, so a correction is never masked by a stale cache hit regardless of TTL length.
- **Background persistence never blocks the response.** Message persistence, session-memory writes, and cache writes run as a fire-and-forget `asyncio.create_task()` after the SSE stream's `done` event, using their own DB session (not the request-scoped one, which FastAPI tears down as soon as the route returns — reusing it caused an intermittent "transaction is closed" race, now fixed).

## Fail-closed ingestion, fail-open UI

- `scripts/seed.py` retries a failed ingest up to 3 times (60s, then 120s backoff, sized to typical OpenAI rate-limit reset windows). If all 3 fail, it exits 1 and **the backend container never starts** (`depends_on: seed: condition: service_completed_successfully`) — the system refuses to serve queries against a partially or incorrectly ingested dataset.
- The frontend container is **not** gated on the backend's readiness (only ordered to start after it, via plain `depends_on`). If the backend never comes up, the React app still loads and renders normally; only the login form's `POST /auth/token` call fails.

## Per-restaurant login and JWT issuance

- Two-key model: the shared `API_KEY` (shipped to the browser — Vite inlines `VITE_*` vars at build time) only proves "this is a legitimate client app," never which restaurant. `POST /auth/token` additionally requires a per-restaurant `restaurant_key` that must match a SHA-256 hash stored in `restaurant_credential` (`src/models/db_entities.py`) before it mints a JWT scoped to that `restaurant_id` (`src/api/routes/auth.py`, `src/utils/restaurant_auth.py`) — closing a real gap where the shared key alone used to be enough to mint a token for *any* restaurant. Same 401 for an unknown `restaurant_id` and a wrong key, so an unauthenticated caller can't enumerate valid restaurant IDs.
- Provision/reissue a restaurant's key with `python scripts/create_restaurant_credential.py <restaurant_id>` (prints the plaintext once, never persisted); `scripts/seed.py` auto-provisions one for the dev restaurant on first run (idempotent — won't rotate an existing one).
- The frontend's old free-pick-any-restaurant dropdown (`RestaurantSelector.tsx`) was replaced with a real login form (`LoginPage.tsx`): restaurant ID + access key, gating the whole app behind a successful `POST /auth/token` call.

## Corrections: unverified flag vs. consensus

- `find_correction()` (`src/core/correction.py`) returns a `CorrectionMatch(text, is_consensus)`, not a bare string. `CONSENSUS_THRESHOLD = 3`: a correction flagged by fewer than 3 distinct sessions is surfaced to generation as `unverified_note` — mentioned briefly if relevant, but explicitly instructed not to override review evidence. Once a correction reaches the threshold, `is_consensus=True` and it's surfaced as `corrections` instead — the existing "supersedes conflicting evidence" instruction, unchanged. This directly closes an abuse path found while designing it: a single flag (e.g. from a competitor spamming a rival's guessed `restaurant_id`) can no longer permanently override real evidence on its own.

## Cross-session memory

- `build_session_context()`'s relevant-turn search (`src/core/session.py`) is filtered by `restaurant_id`, not `session_id` — a relevant exchange from a *different*, earlier conversation with the same restaurant surfaces here too, not just turns from the current session. A cross-session turn is labeled with how long ago it happened ("from a past conversation, N day(s) ago") so the model can judge staleness, and `MAX_CROSS_SESSION_AGE_DAYS = 90` excludes anything older outright — a deliberate safeguard, since the old session-scoped search was naturally bounded by one conversation's lifetime and this one isn't.

## Vague and ungroundable-metric questions

- `build_retrieval_params()` (`src/core/retrieval.py`) deterministically forces `needs_aggregation=True` (and therefore `top_k=20` instead of 6) whenever `intent == "improvement"`, rather than relying on the model to infer that a broad "how can I improve?" needs wider retrieval on its own.
- The improvement-suggestion prompt rule (`chat_response_complex.yaml`) is three-way, not binary: strong evidence on a topic suggests the specific fix it implies; thin/scattered evidence states the real themes found *first*, then may add clearly-labeled general best-practice advice (never blended in as if it were evidence-backed); zero evidence still hits the existing hard hallucination gate upstream.
- Both generation prompts gained a guardrail against inventing a causal number connecting reviews to a business metric they don't contain (sales, revenue, foot traffic) — the model states plainly that reviews can't measure it, optionally offering sentiment/volume trends as a directional proxy instead.

## Resumable batch ingestion and live single-review ingestion

- `_process_rows` (`src/workers/ingest_worker.py`) writes to Postgres in the same per-batch loop as the Qdrant upsert, not in one pass after the whole file finishes — a prior version deferred the Postgres write entirely to the end, so a crash partway through left Postgres empty regardless of how much Qdrant already had. With both stores committing together per batch, a `review_id` + `pipeline_version` skip check (`_fetch_processed_review_ids`) is now a reliable resumability signal: a retried job never re-chunks, re-extracts entities, or re-embeds a review a prior attempt already finished.
- `POST /api/v1/ingest/review` (`ingest_single_review()`) is the push-based counterpart: a source system can call it the moment one review posts or is edited, instead of waiting for a full batch re-upload. `review_id` is derived from `(restaurant_id, external_review_id)`, so a repeat call with the same ID is a genuine update — it reuses the same chunk_ids and explicitly deletes any stale leftover chunks (from both Qdrant and Postgres) if an edit shrinks the review's chunk count, since `ON CONFLICT DO UPDATE` alone never touches those orphans.

## Answer quality: staleness signal and quantified claims

- **Staleness caveat, suppressed per-restaurant not per-message.** `rank_results()` (`src/core/ranking.py`) flags an answer when most retrieved evidence predates `DATA_STALENESS_DAYS` (365). For a fixed historical dataset this is true of nearly every query — real time keeps moving forward while the corpus doesn't — so repeating it on every single response was pure noise. The backend signal is unchanged (still computed fresh, still occasionally meaningful); the frontend (`chatStore.ts`) suppresses repeats by persisting "already shown" per `restaurant_id` in `localStorage`, not per session — a per-session version was tried first but a new conversation reset it every time, defeating the point.
- **Generation must cite real numbers, not vague quantifiers.** `entity_counts` (per-dish/item mention tallies across retrieved evidence) was already passed into the complex generation prompt but nothing told the model to actually use it. Live-observed failure: an answer said reviewers "frequently mention" a complaint instead of citing the real count. `chat_response_complex.yaml`'s rule now requires citing the actual number (`entity_counts`, `exact_count`, or a real count of the evidence shown) wherever one is available, instead of "some"/"several"/"many".

## Decomposition robustness: context scoping, sentiment safety net, conversation recall

Three related bugs surfaced through live testing after the fixes above, all rooted in decomposition (a small, fast, free-tier model) being trusted a little too far for things that are cheaply checkable in code:

- **Session context must not bleed into intent classification.** `session_context` (`src/core/decomposition.py`, fed for pronoun/reference resolution) was observed pulling a clear, self-contained follow-up ("total positive reviews," right after two turns about improvement/complaints) toward `intent=improvement` instead of `count_query` — confirmed via `logs/request_traces.jsonl` showing the wrong intent and a 20-chunk retrieval instead of an instant DB count. `query_decomposition.yaml` now explicitly scopes session context to reference resolution only; it must never override classification of an already-unambiguous new question.
- **Count queries get a deterministic sentiment-polarity safety net.** Separately observed: "total negative reviews" came back with `sentiment_filter=Positive` — decomposition extracted the wrong polarity on an unambiguous query. Since a `count_query`'s entire value is an exact, trustworthy number, `_resolve_sentiment_filter()` (`src/api/routes/chat.py`) overrides decomposition's extraction with a literal keyword match whenever the raw query names exactly one sentiment word unambiguously (falls back to decomposition's own extraction if zero or multiple keywords match, e.g. "compare positive and negative reviews").
- **`conversation_recall` is its own intent, not a research question.** "What did I ask before?" was first misclassified `ui_question` (wrongly redirected to CareBot, which has no visibility into this chatbot's own history), then reclassified `factual` — which fixed the redirect but still ran full review retrieval, attaching irrelevant evidence, a meaningless confidence score, and the staleness caveat to an answer that has nothing to do with reviews. It's now a dedicated intent (`src/api/routes/chat.py`, `prompts/files/conversation_recall.yaml`) that skips retrieval entirely and answers from session context alone: no evidence, no fabricated confidence, no caveat. Its prompt also has to be explicit that session context mixes turns from *this* conversation with age-labeled turns from *other* past ones (cross-session memory) — live-caught blending a different session's topic into "in this session, you asked about X" is a factual error the prompt now has to actively guard against.
- **Debug-level prompt logging exists specifically so this class of bug is verifiable without reconstructing it from trace + DB records after the fact.** Both `decompose_query()` and `build_generation_prompt()` log the fully-assembled system/user prompt (session context included) at `LOG_LEVEL=DEBUG`.

## Insights report charts

- `ReportCharts.tsx` (Recharts) surfaces the same structured numbers `generate_report()` already computed (rating distribution, sentiment/source breakdowns, top praised/complained entities) as an actual rating histogram, sentiment pie chart, platform breakdown, and top-praised-vs-complained bars, rendered alongside the existing markdown narrative in `ReportView.tsx` — no backend change needed, the data was already there.

## Known limitations / deviations from the standard

- **CI coverage gate is a 50% floor, not the 80% target.** `src/api/routes/*` and most of `src/services/llm|vector` are only exercised by the live e2e eval harness (needs Postgres/Qdrant/Redis + real LLM keys), which CI intentionally excludes. Closing the gap needs FastAPI `TestClient` + `dependency_overrides` route tests, which don't need live infra — not yet written. Documented inline in `.github/workflows/ci.yml`.
- **The eval fixture's simulated-Qdrant-outage case (RB-01) is based on a now-stale assumption.** It expects the system to "degrade gracefully to BM25-only retrieval" if Qdrant goes down — but since sparse (BM25-style) vectors moved server-side into Qdrant itself, a Qdrant outage now takes down both dense and sparse retrieval together; there is no independent in-process BM25 fallback anymore. The eval harness documents this as an intentionally skipped case rather than faking a pass.
- **Deployment: production Dockerfiles exist (`infra/docker/backend`, `infra/docker/frontend`), but no cloud target is chosen yet.** See `docs/runbook.md` for the build/push procedure and what's still needed to actually deploy somewhere.
