Solid arrows show control flow; dashed arrows show data reads/writes or provider calls. docker-compose brings up infrastructure, runs the seed service once, and only starts the backend on service_completed_successfully. The seed checks the ingest_manifest table (hash + PIPELINE_VERSION) and exits in under a second on a match; on a mismatch or first run it calls run_ingest_job — the same function behind POST /ingest — with up to three attempts and 60s x attempt backoff.

If seed exhausts all 3 attempts (SD9, EXIT 1): the backend container never starts (its service_completed_successfully dependency is never satisfied). The frontend container, however, is NOT gated on the backend's readiness — it starts independently and serves the React app normally. A user opening the app sees the UI shell and RestaurantSelector load fine, but GET /restaurants fails; the selector's isError state renders "Server unavailable. Is the backend running?" with a Retry button rather than an infinite spinner or a cryptic network error. This is a deliberate fail-closed design at the data layer (never serve queries against a partially/incorrectly ingested dataset) paired with a fail-open UI shell (the frontend itself never crashes just because the backend isn't up yet).
